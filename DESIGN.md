# 擦水晶 · Wipe the Crystal — 理念文档

> 一款轻松解压的 3D 擦拭小游戏。Godot 4.6 / gl_compatibility，一套代码同发 iOS 与 Web(GitHub Pages)。
> 场景全部脚本生成(`main.gd`),无 `.tscn`。

---

## 一、核心理念

### 1. 一句话
你是小镇上的**解谜大师**。为查清近来的流言,你翻出箱底那盒尘封的**水晶拼图**,喷水擦去灰尘,让蒙尘的水晶重现通透光泽——随着灰尘褪去、光芒透出,真相也逐渐浮现。

### 2. 情绪目标:解压
- 无失败、无倒计时、无强目标。玩家自由地擦。
- 反馈要**即时、柔和、可控**:擦到哪、哪里就从灰变亮;水声、水花、光芒都是正反馈。
- "干净"本身是奖励。擦净约 99.9% 可"交付",一声清脆铃音收尾。

### 3. 核心循环
`覆尘水晶 → 喷水擦拭露出通透水晶 → 擦净 99.9% → 交付(奖励音) → 刷新换一个随机水晶重来`

---

## 二、玩法设计原则

- **单指=擦拭,双指=看**:单指按住/拖动喷水冲刷;双指捏合缩放 + 平移旋转;拖空白背景也可旋转。职责清晰不打架。
- **擦拭是"揭示"而非"破坏"**:灰尘不是被"削掉"的实体,而是一层遮罩;擦拭把遮罩擦开,露出本就存在的光滑水晶。这比"体素粉碎"更贴合"擦"的直觉,也更省。
- **随机与新鲜感**:每次刷新随机换一个水晶模型、随机的初始无尘擦痕、随机的入场旋转轴与角度(4 整圈 + 额外随机 0~180°,停在随机朝向)。
- **初始态要"像真擦过"**:开局的无尘区不是撒点,而是几笔**隐藏的随机擦拭**(真实笔刷沿表面走连贯路径)做到约 20% 无尘——看起来是"擦痕"而非"斑点"。
- **状态持久化**:擦拭进度、当前模型、按钮态、日夜设置都存 `user://`,退出再进接着玩。解压游戏不该丢进度。
- **交付是仪式**:达标后右上角按钮变圆圈(一声短"叮"提示),点击交付变对勾 + 奖励音,并锁定擦拭只留旋转欣赏;再点重来。

## 三、视觉/手感理念

- **纯黑背景 + 发光水晶**:突出主体,神光、反光在黑底上最亮眼。
- **灰尘哑光、水晶通透**:灰尘是纯漫反射的暗哑粉末;露出的水晶半透明、强反光、带蓝、能透看背面刻面。二者反差就是"擦"的爽点。
- **丁达尔/神光**:水晶的强反光高光通过屏幕空间体积光后处理(crepuscular rays)拉成光柱,在黑底空气中散射,像光透过水晶。
- **克制的粒子**:喷水水花是点缀不是主角——恒定透明度、松手整体淡出、数量与亮度都压在"看得见但不喧宾夺主"。
- **日/月两态**:☀️ 暖亮 ↔ 🌙 冷暗,一键切换整体光照与色温。

## 四、技术架构与关键取舍

目标渲染后端是 **gl_compatibility**(Web + 老 iOS 都能跑),很多"高级"特性不可用,于是大量用"够用且可控"的手法替代。

| 需求 | 方案 | 为什么 |
|---|---|---|
| 擦拭揭示,覆盖无上限 | **UV 遮罩贴图**(1024²,L8):擦拭画进遮罩,shader 按 `UV` 采样混合 | 最初用"最多 80 个冲刷点数组"会用尽、擦不干净;UV 遮罩无上限、边缘细腻,UV 分岛天然只作用命中那一面 |
| 灰尘要实、水晶要透 | **双趟材质(next_pass)**:灰尘层不透明写深度、剔背面;水晶层透明双面不写深度,各自按遮罩 `discard` 对方区域 | 单材质做不到"一面实一面透";两趟解决旋转穿模,又保留水晶通透可见刻面 |
| 折射丁达尔 | **屏幕空间体积光后处理**(CanvasLayer + `hint_screen_texture` 径向模糊),而非真体积雾 | Compatibility 无体积雾;后处理在纯黑背景下效果好、可控 |
| 强反光(黑底无天空可反射) | shader 里**程序化"影棚"环境反光**(按反射方向生成亮斑)+ 边缘辉光 + 泛光 | 纯金属反光会反射成黑色;伪环境反光在黑底也能强反光 |
| 命中拾取擦拭点 | 射线命中 → 找**最近顶点的 UV** 画进遮罩 | 物理射线拿不到 UV;最近顶点法足够;还顺带用最近顶点的**着色法线**做"只擦命中面"判定(避免某些面法线被翻反而擦不掉) |
| 安全区 | 读 `DisplayServer.get_display_safe_area()`,按画布/屏幕缩放换算,放置右上/左上按钮 | 避让刘海/灵动岛/圆角 |
| 覆盖率/交付判定 | 统计"遮罩已露出的顶点占比" | 近似表面无尘比例,稳定可控;初始擦拭也用它来做到目标 20% |

### 光照
主平行光 + 前上方聚光 + 水晶中心一盏内部 OmniLight(透过露出处透内芒)+ 环境光;日夜切换统一调色温与能量。

## 五、工程踩坑与经验(尤其"水花光球")

这段是给后来者的真实教训,值多看一眼:

1. **屏幕空间后处理会"看到"一切**。丁达尔神光读屏时会把**任何**足够亮的像素(包括叠加的水珠)拉成光柱/光球。→ 想排除某物,得让它**在后处理之后合成**。
2. **半透明 MIX 叠加会累积亮度**。很多 alpha 0.08 的水珠重叠,会累积成接近不透明的亮/暗团。→ 别让粒子在一点堆积(减阻尼让其飞散、限制数量、或让它不参与后处理)。
3. **`SubViewport.own_world_3d` 默认 false = 共享主世界**。把粒子丢进子视口"以为隔离了",其实它还在主世界里带深度渲染。这正是那个"能被灰尘按深度遮挡、附着接触面、滞留 1 秒"的光球真凶。→ 要真正隔离,必须 `own_world_3d = true`。
4. **单槽位粒子(amount=1)回收有一帧空档**:新粒子必须等旧的死亡释放槽位才生成,淡入淡出补不上缝。→ 用 ≥2~3 个槽位让循环重叠;或干脆**不做逐颗淡入淡出**,改成"擦时恒定、松手时整体材质透明度 tween 到 0"。
5. **诊断法胜过猜测**:定位光球时,靠逐个"临时关掉某项(水花/内光/自发光/镜面)看它还在不在"来二分,比反复猜快得多。用户的"它能被灰尘按深度遮挡"这类观察一句顶十句。
6. **JSON/GLB 打包**:`export_filter=all_resources` 不打包普通 `.json`,需在 `include_filter` 加 `*.json`;GLB 离线体素化/解析用 numpy 手写(无 trimesh)。

## 七、演进史(为什么它长这样)

这个工程是**一路重构**过来的:桌板令牌游戏(小魔典)→ 3D 拼图 → 宝石切割(体素粉碎)→ 覆粉钻石冲刷 → 最终定型为"擦水晶"。核心教训:**当机制和直觉/性能反复打架时,换一个更贴合直觉又更省的表达**(体素粉碎 → UV 遮罩揭示),往往比死磕更好。名字也随之从"小魔典"→"解谜小宝石"→"擦水晶"。
